# 陇上管家 (longarch2) · 部署上线指南

> 面向第一次部署的同学：照着一步步做即可。命令可以直接复制粘贴，遇到不懂的名词可以先按默认值走。
>
> 整套系统分三块：
> - **longarch-server**：后端（Spring Boot，端口 8081）
> - **longarch-admin-next**：管理后台网页（React 19 + Vite，端口 8080）
> - **miniapp**：微信小程序（Taro 4，编译后上传到微信平台，不进服务器）
>
> MySQL / Redis / MQTT / SRS 等依赖都由 `docker compose` 一起拉起来，不用单独装。

---

## 0. 先准备这些东西

| 必备项 | 说明 |
| --- | --- |
| 一台云服务器 | 推荐 2 核 4G 起，系统 Ubuntu 22.04。需要有公网 IP。 |
| Docker + Docker Compose | 安装方式见下方 1.1 |
| 一个域名 | **正式上线必须**（微信小程序只允许 HTTPS 域名访问，且登录 Cookie 在生产下只走 HTTPS） |
| 微信小程序的 AppID / AppSecret | 在微信公众平台 → 开发 → 开发设置里拿 |

可选（用到才需要）：
- 物联网设备 / MQTT（用于远程下发浇水施肥指令）
- 智谱 GLM AI Key（用于 AI 助手 / 地块分析功能）
- 摄像头 + SRS 流媒体（用于摄像头直播，已编排进 compose）

## 1. 第一种部署：演示 / 内测版（最快，5 分钟跑起来）

> 适合：先把系统跑起来看效果、内部演示。
> 特点：用「模拟微信登录」，自动创建一个管理员账号，开着接口文档。
> ⚠️ **不安全，不要直接对公网开放当正式环境用**（真实微信登录也用不了，需要看第 2 章）。

### 1.1 安装 Docker（服务器上执行一次）

```bash
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker $USER     # 让当前用户能用 docker，执行后重新登录一次
```

### 1.2 拉代码并启动

```bash
git clone https://github.com/minyudin/longarch2.git
cd longarch2
cp .env.example .env              # 演示版用默认值即可，不用改
docker compose up -d --build      # 第一次会构建镜像，耐心等几分钟
```

### 1.3 确认启动成功

```bash
docker compose ps                                   # 6 个服务都应是 Up / healthy
curl http://localhost:8081/actuator/health          # 返回 {"status":"UP"} 即后端就绪
```

### 1.4 登录管理后台

- 浏览器打开：`http://你的服务器IP:8080`
- 账号 / 密码：`15675201507` / `admin123456`

到这一步演示版就完成了。想看接口文档：`http://你的服务器IP:8081/doc.html`。

## 2. 第二种部署：正式上线版（生产）

> 适合：真正对外开放、真实微信登录。
> 生产模式下系统会启动「安全守卫」（ProductionStartupGuard），**少配一项就拒绝启动**。下面按顺序把每一项配好即可。

整体顺序：
1. 先用一次「演示模式」把管理员账号建好（顺便建好数据库字段）
2. 填好生产环境变量 `.env`
3. 配好 MQTT（生产强制要求，见 2.4）
4. 切到生产模式启动后端
5. 用域名 + HTTPS 把网站暴露出去
6. 编译并上传微信小程序

---

### 2.1 建管理员账号（只做一次）

生产模式**不会**自动创建管理员，所以我们先借「演示模式」把账号和数据库字段建好（数据会存在 MySQL 数据卷里，之后切生产仍然在）。

编辑 `.env`，把管理员手机号和密码改成你自己的：

```bash
# .env 里这几行（没有就加上）
SPRING_PROFILES_ACTIVE=dev
ADMIN_SEED_MOBILE=你的手机号
ADMIN_SEED_PASSWORD=你的强密码
```

启动一次，让它把管理员写进数据库：

```bash
docker compose up -d --build
docker compose logs server | grep -i seed   # 看到播种成功日志即可
```

确认后继续下一步（先不用关，下面会重启切生产）。

---

### 2.2 填写生产环境变量 `.env`

把 `.env` 改成生产配置。下面是**最少必填**项：

```bash
# ---- 切换到生产模式 ----
SPRING_PROFILES_ACTIVE=prod

# ---- 数据库 / Redis（改成强密码）----
DB_PASSWORD=换成你的数据库强密码
REDIS_PASSWORD=换成你的Redis强密码

# ---- 微信小程序（必填真实值，不能是 stub 占位）----
WX_APP_ID=你的真实AppID
WX_APP_SECRET=你的真实AppSecret

# ---- 大屏访问令牌（必填，随便一段长随机字符串）----
DASHBOARD_TOKEN=换成一段长随机字符串

# ---- MQTT（见 2.4 节）----
# 方案①最省事：不用任何证书，明文 tcp://（仅限 broker 在内网/同机、外部连不到时）
MQTT_BROKER_URL=tcp://mosquitto:1883
MQTT_AUTH_MODE=password
MQTT_USERNAME=换成你的MQTT账号
MQTT_PASSWORD=换成你的MQTT密码
MQTT_ALLOW_INSECURE=true
# 方案②加密：把上面几行换成下面的（broker 走 TLS），见 2.4 方案②
# MQTT_BROKER_URL=ssl://你的MQTT地址:8883
# MQTT_AUTH_MODE=password
# MQTT_ALLOW_INSECURE=false
# MQTT_SSL_TRUST_STORE=/certs/truststore.p12
# MQTT_SSL_TRUST_STORE_PASSWORD=你的truststore密码

# ---- 智谱 AI（可选，不填则 AI 功能不可用，但不影响启动）----
ZHIPU_API_KEY=
```

> 生成随机令牌的小技巧：`openssl rand -hex 32`

生产守卫会检查的项（少了就报错，见第 4 章对照表）：
- `SPRING_PROFILES_ACTIVE=prod`（prod profile 下 `wechat.miniapp.stub-mode` 已强制为 false）
- 真实 `WX_APP_ID` / `WX_APP_SECRET`（不能是 stub 占位）
- `DASHBOARD_TOKEN` 非空
- MQTT 地址不能是 localhost；默认必须是 `ssl://`，**除非设了 `MQTT_ALLOW_INSECURE=true` 才允许明文 `tcp://`**
- MQTT 账号密码不能用 dev 默认值（`longarch`/`longarch123`）
- `mqtt.strict-device-identity` / `require-message-id` / `require-timestamp` 在 prod 下必须为 true（`application-prod.yml` 已默认 true，无需手动设）
- 管理员自动播种在生产下默认关闭（无需手动设置）

### 2.3 关掉对公网暴露的内部端口（安全）

`docker-compose.yml` 默认把 MySQL(3306)、Redis(6379)、MQTT(1883)、SRS(1935/1985/8880) 也映射到了主机端口，方便本地调试。**正式上线建议只对外暴露 8080（网站）和 8081（后端），其余只在内网/本机可达**。最简单的办法是用服务器防火墙/安全组只放行 80、443 端口（见 2.7 用 HTTPS 后，连 8080/8081 都可以不对外开放）。

---

### 2.4 配置 MQTT

两种方案，按需二选一：

#### 方案①（推荐 / 最省事）：不用证书，明文 tcp://

如果你的 MQTT broker 和后端跑在同一台服务器（或同一内网），外部公网连不到它，就**不需要任何 TLS/证书**。在 `.env` 里这样配即可：

```bash
MQTT_BROKER_URL=tcp://mosquitto:1883
MQTT_AUTH_MODE=password
MQTT_USERNAME=换成你的MQTT账号      # 不能用 dev 默认的 longarch
MQTT_PASSWORD=换成你的MQTT密码      # 不能用 dev 默认的 longarch123
MQTT_ALLOW_INSECURE=true            # 关键：允许明文 tcp://
```

> `MQTT_ALLOW_INSECURE` 默认是 `false`（强制 `ssl://`）。只有你确认 broker 不对公网开放时才设为 `true`。compose 自带的 `mosquitto` 容器默认就是 `tcp://mosquitto:1883`，可直接用。⚠️ 切勿把 1883 端口对公网开放。
> 注意：`application-prod.yml` 的 MQTT 默认 `auth-mode=mtls`，走方案① 时**必须显式设 `MQTT_AUTH_MODE=password`**，否则守卫会要求你提供客户端证书。

配好后跳到 2.5（摄像头）或 2.6（启动）即可，下面方案②可以忽略。

#### 方案②（加密）：MQTT over TLS

如果 broker 需要对公网暴露，则建议走加密连接（`ssl://`），不要设 `MQTT_ALLOW_INSECURE`。

- **如果你暂时没有物联网设备**：可以按下面的「自建 TLS Mosquitto」做一个最简单的。
- **如果你已经有 MQTT 服务商/Broker**：直接把 2.2 里的 `MQTT_*` 改成它的 `ssl://` 地址和账号密码即可。

#### 自建 TLS Mosquitto（最简版，密码认证）

在服务器上生成自签证书：

```bash
mkdir -p ~/mqtt-certs && cd ~/mqtt-certs
# 1) 生成 CA
openssl req -new -x509 -days 3650 -nodes -keyout ca.key -out ca.crt -subj "/CN=longarch-ca"
# 2) 生成服务器证书（CN 必须是你 MQTT 的域名或 IP）
openssl req -new -nodes -keyout server.key -out server.csr -subj "/CN=你的MQTT域名或IP"
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 3650
# 3) 把 CA 打包成后端用的 truststore（记住这里的密码）
keytool -importcert -noprompt -alias longarch-ca -file ca.crt \
  -keystore truststore.p12 -storetype PKCS12 -storepass 你的truststore密码
```

设置 Mosquitto 密码文件：

```bash
docker run --rm -v ~/mqtt-certs:/c eclipse-mosquitto:2 \
  mosquitto_passwd -b -c /c/passwd 你的MQTT账号 你的MQTT密码
```

把 `deploy/mosquitto/mosquitto.conf` 换成 TLS 版（端口 8883）：

```conf
listener 8883
allow_anonymous false
password_file /mosquitto/config/passwd
cafile   /mosquitto/certs/ca.crt
certfile /mosquitto/certs/server.crt
keyfile  /mosquitto/certs/server.key
persistence true
persistence_location /mosquitto/data/
```

让证书能被后端容器读到（把 truststore 挂进 server）。在 `docker-compose.yml` 的 `server` 和 `mosquitto` 服务里挂载 `~/mqtt-certs`：

```yaml
  mosquitto:
    ports:
      - "8883:8883"
    volumes:
      - ./deploy/mosquitto/mosquitto.conf:/mosquitto/config/mosquitto.conf:ro
      - /home/你的用户/mqtt-certs:/mosquitto/certs:ro
      - /home/你的用户/mqtt-certs/passwd:/mosquitto/config/passwd:ro
      - mosquitto-data:/mosquitto/data

  server:
    volumes:
      - /home/你的用户/mqtt-certs:/certs:ro      # 对应 .env 里的 MQTT_SSL_TRUST_STORE=/certs/truststore.p12
```

然后 `.env` 里：
```bash
MQTT_BROKER_URL=ssl://mosquitto:8883
MQTT_AUTH_MODE=password
MQTT_USERNAME=你的MQTT账号
MQTT_PASSWORD=你的MQTT密码
MQTT_SSL_TRUST_STORE=/certs/truststore.p12
MQTT_SSL_TRUST_STORE_PASSWORD=你的truststore密码
```

### 2.5 配置摄像头直播（SRS，可选）

> 没有摄像头需求可跳过本节，不影响其它功能启动。

摄像头直播用的是 **SRS 媒体服务器**，已经作为 `srs` 服务编排进 `docker-compose.yml`（镜像 `ossrs/srs:5`，配置在 `deploy/srs/srs.conf`），`docker compose up` 时会自动一起启动，**无需单独安装**。

协议与端口：
- 推流（摄像头/设备 → SRS）：**RTMP**，端口 `1935`，推到 `rtmp://<服务器>:1935/live/<摄像头编号>`
- 播放（SRS → 前端）：**HTTP-FLV**（端口 `8880`，主用，低延迟）`http://<服务器>:8880/live/<摄像头编号>.flv`；HLS（`.m3u8`，备用）
- 管理 API（端口 `1985`）：后端用它判断摄像头是否在推流

后端通过下面 4 个环境变量找到 SRS（已在 compose 配好默认值）：

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `SRS_API_BASE` | `http://srs:1985` | 后端容器内部访问，**用服务名 `srs`，不要改成 localhost** |
| `SRS_HTTP_FLV_BASE` | `http://localhost:8880` | 返回给浏览器/小程序的 FLV 播放地址，**上线必须改成你的域名** |
| `SRS_HLS_BASE` | `http://localhost:8880` | 同上（HLS 备用） |
| `SRS_RTMP_BASE` | `rtmp://localhost:1935` | 推流地址提示 |

**演示/内网**：默认值即可，浏览器在同机用 `localhost:8880` 能直接播。

**正式上线**：FLV/HLS 地址要让用户的浏览器能访问到，必须在 `.env` 里改成你的域名，并让它能走 HTTPS（建议像 `/api` 一样用 Caddy/nginx 反代一个路径到 `srs:8880`，避免 http 混内容被浏览器拦截）：

```bash
SRS_HTTP_FLV_BASE=https://你的域名/live-flv
SRS_HLS_BASE=https://你的域名/live-flv
```

> ⚠️ RTMP 的 1935 端口只给内网设备推流用，**不要无脑对公网开放**；如需远程推流请加鉴权或限制来源 IP。

推一路测试流验证（服务器上装了 ffmpeg 即可，摄像头编号换成你后台里的设备号）：

```bash
ffmpeg -re -f lavfi -i testsrc=size=640x360:rate=25 -c:v libx264 -preset veryfast -f flv \
       rtmp://localhost:1935/live/你的摄像头编号
# 另开一个终端拉流, 返回内容前 3 字节是 FLV 就说明通了
curl -s http://localhost:8880/live/你的摄像头编号.flv -o /tmp/t.flv && head -c3 /tmp/t.flv
```

推流后，后台「摄像头」里该摄像头会自动变为「在线/推流中」，前端调 `/api/v1/cameras/{id}/live-url` 即可拿到 FLV 播放地址。

### 2.7 配置域名 + HTTPS（必须）

生产下登录 Cookie 只在 HTTPS 下才会下发，所以**必须用域名 + HTTPS**访问。最省心的方式是加一个 Caddy 容器，自动申请 Let's Encrypt 证书。

先把你的域名解析到服务器公网 IP（A 记录），然后：

```bash
cd longarch2
cat > Caddyfile <<'EOF'
你的域名.com {
    reverse_proxy admin:80
}
EOF

docker run -d --name caddy --restart unless-stopped \
  --network longarch2_default \
  -p 80:80 -p 443:443 \
  -v $(pwd)/Caddyfile:/etc/caddy/Caddyfile:ro \
  -v caddy_data:/data \
  caddy:2
```

> 网络名一般是 `longarch2_default`，可用 `docker network ls` 确认。
> Caddy 会把 `https://你的域名.com` 转发到管理后台容器，后台再把 `/api` 转发给后端，全程同源，登录 Cookie 正常。

完成后用 `https://你的域名.com` 打开管理后台，用 2.1 设的账号密码登录。

### 2.8 编译并上传微信小程序

小程序要单独编译上传，并且要切到「真实微信登录」模式（和后端对齐）。

在本地电脑（不是服务器）上：

```bash
cd miniapp
npm i -g pnpm
pnpm install
# 关键：AUTH_MODE=wechat 走真实登录；API_BASE 指向你的 HTTPS 域名
TARO_APP_AUTH_MODE=wechat TARO_APP_API_BASE=https://你的域名.com/api/v1 pnpm build:weapp
```

然后：
1. 用「微信开发者工具」打开 `miniapp/dist/weapp` 目录，填入你的小程序 AppID。
2. 在微信公众平台 → 开发 → 开发设置 → **服务器域名** 里，把 `https://你的域名.com` 加到 `request 合法域名`。
3. 在开发者工具里点「上传」，再到公众平台提交审核、发布。

> 不加 `TARO_APP_AUTH_MODE=wechat` 的话，小程序会继续用「演示模式」的设备号当登录凭证，真实微信登录会失败。

## 4. 启动报错对照表（生产守卫）

后端启动失败、日志里有 `Production configuration invalid:` 时，按提示找对应行：

| 报错关键字 | 原因 | 怎么改 |
| --- | --- | --- |
| `wechat.miniapp.stub-mode must be false` | 还在演示模式 | `.env` 设 `SPRING_PROFILES_ACTIVE=prod` |
| `wechat.miniapp.app-id / app-secret must be real values` | 微信 AppID/Secret 没填或是 stub 占位 | 填真实 `WX_APP_ID` / `WX_APP_SECRET` |
| `dashboard.token must be configured` | 大屏令牌为空 | 设 `DASHBOARD_TOKEN`（用 `openssl rand -hex 32` 生成） |
| `mqtt.broker-url must use ssl:// in prod` | MQTT 用了 tcp:// 但没开放行开关 | 内网 broker 设 `MQTT_ALLOW_INSECURE=true`（方案①）；或改成 `ssl://...:8883`（方案②）。见 2.4 |
| `mqtt username/password must not use development defaults` | 还在用 dev 默认账号密码 | 把 `MQTT_USERNAME`/`MQTT_PASSWORD` 改成非 `longarch`/`longarch123` 的值 |
| `mqtt.broker-url must not point to localhost` | MQTT 指向了本机 | 用服务名 `mosquitto` 或真实地址 |
| `mqtt.username/password must be configured when mqtt.auth-mode=password` | password 模式缺账号密码 | 补 `MQTT_USERNAME` / `MQTT_PASSWORD` |
| `mqtt.ssl.key-store / trust-store must be configured when mqtt.auth-mode=mtls` | mtls 模式缺证书 | 改用 `MQTT_AUTH_MODE=password`（方案①），或补 `MQTT_SSL_KEY_STORE` / `MQTT_SSL_TRUST_STORE` |
| `longarch.admin.seed.enabled must be false in prod` | 误开了管理员播种 | 删掉 `.env` 里的 `LONGARCH_ADMIN_SEED_ENABLED=true` |

## 6. 各服务 / 端口速查

| 服务 | 容器端口 | 默认对外端口 | 说明 |
| --- | --- | --- | --- |
| admin（管理后台） | 80 | 8080 | 网页入口，内部反代 `/api` 到后端 |
| server（后端） | 8081 | 8081 | REST API + 健康检查 `/actuator/health` |
| mysql | 3306 | 3306 | 数据库（生产建议别对公网开放） |
| redis | 6379 | 6379 | 缓存（生产建议别对公网开放） |
| mosquitto | 1883 / 8883 | 1883 / 8883 | MQTT（生产用 8883 TLS，或方案①明文 1883 仅限内网） |
| srs | 1935 / 1985 / 8880 | 1935 / 1985 / 8880 | 摄像头直播：RTMP 推流 / 管理 API / HTTP-FLV·HLS 播放 |

> 端口都可以在 `.env` 里改（`SERVER_PORT` / `ADMIN_PORT` / `DB_PORT` / `SRS_*_PORT` 等）。
