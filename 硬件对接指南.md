# 陇上管家 · 硬件对接指南

> 面向真实设备(STM32 / ESP32 / 树莓派 / 网关)的最小对接契约。照着实现,就能接入平台完成"下发指令 → 执行 → 回执 → 失败兜底"的完整闭环。

---

## 1. 目的

把物理执行器(浇水泵、施肥阀、喷淋头等)接入陇上管家平台,让用户能:

- 从微信小程序点一下"浇水 5 分钟",指令能准确送到设备
- 看到设备**是否真的执行成功**,而不是"命令已发送"
- 设备异常时,平台能在**几秒内**判失败并告知用户
- 平台能感知设备是否在线,离线时自动禁用相关操作

## 2. 适用范围

| 硬件形态 | 是否推荐 | 说明 |
| --- | --- | --- |
| ESP32 + 继电器 + 水泵 | ⭐️⭐️⭐️ **首选** | 单片就是完整设备,WiFi+MQTT+TLS 都是现成组件 |
| STM32 + WiFi 模块 + 继电器 | ⭐️⭐️ | 可行但吃紧,TLS 握手时堆内存要抠 |
| STM32 群 + 网关(ESP32/树莓派) | ⭐️⭐️⭐️ **生产推荐** | 一路 WiFi 挂多个 RS485/CAN 设备,布线简单 |
| 工控机 / PLC 网关 | ⭐️⭐️⭐️ | 大棚级方案,可跑复杂本地逻辑 |

> 平台不假设硬件用哪种芯片,只约定 **MQTT 主题 + JSON payload** 的通信契约。硬件端用什么语言、什么架构都可以。

## 3. 对接总览(设备只需做 3 件事)

```
┌─ 平台(server) ─┐          ┌─ MQTT broker ─┐          ┌─ 你的设备 ─┐
│                │  publish │                │  push    │            │
│  下发指令       │─────────→│ command topic  │─────────→│ 收到 → 执行 │
│                │          │                │          │            │
│                │  subscr. │                │  push    │            │
│  收回执         │←─────────│ callback topic │←─────────│ 上报 ACK    │
│                │          │                │          │ 上报 Result │
│                │  subscr. │                │  push    │            │
│  收心跳         │←─────────│ heartbeat topic│←─────────│ 定时发心跳  │
└────────────────┘          └────────────────┘          └────────────┘
```

设备端要实现的 3 件事:
1. **订阅命令主题** → 收到指令后执行动作
2. **发布回执**(推荐两阶段:ACK + Result)
3. **定期发心跳** → 让平台知道设备还活着

## 4. Topic 契约

命名规则:`longarch/device/{类型}/{设备编号}`

| 用途 | Topic 模板 | 方向 | 举例 |
| --- | --- | --- | --- |
| 下发指令 | `longarch/device/command/{deviceNo}` | server → device | `longarch/device/command/ACT-DM01` |
| 上报回执 | `longarch/device/callback/{deviceNo}` | device → server | `longarch/device/callback/ACT-DM01` |
| 上报心跳 | `longarch/device/heartbeat/{nodeNo}` | device → server | `longarch/device/heartbeat/EDGE-STM32-01` |
| 上报遥测(可选) | `longarch/device/telemetry/{sensorNo}` | device → server | `longarch/device/telemetry/SEN-SOIL-01` |

**MQTT 连接参数**:
- Broker:平台的 Mosquitto,默认端口 `1883`(内网)或 `8883`(TLS)
- 鉴权:账号 + 密码(生产环境)或 mTLS(高安全场景)
- QoS:**1**(至少一次,平台会做去重)
- `cleanSession`:**false**(设备重连能收到断线期间的消息)
- Keep-Alive:60 秒

## 5. Payload 契约

所有消息都是 **UTF-8 JSON**,单条不超过 64 KB。

### 5.1 下发指令(server → device)

Topic:`longarch/device/command/{deviceNo}`

```json
{
  "taskId": 12345,
  "taskNo": "T190123...",
  "deviceId": 80061,
  "deviceNo": "ACT-DM01",
  "actionType": "irrigation_apply",
  "actionParams": {
    "durationMinutes": 5,
    "waterVolumeLiters": 50
  },
  "callbackTopic": "longarch/device/callback/ACT-DM01",
  "timestamp": 1720000000000,
  "expiresAt": 1720000330000
}
```

**设备需要做的**:
- 校验 `deviceNo` 是自己
- 校验 `expiresAt`:如果本地时间已经超过 `expiresAt`,**不要执行**,直接回一条 `failed` 回执并带 `failReason: "expired"`。这是防止"平台已经判超时了,设备还在傻乎乎地开阀门"的双侧不一致
- 解析 `actionType` 和 `actionParams`,打开对应 GPIO / 继电器
- 记住 `taskId` 和 `callbackTopic`,后面要用

### 5.2 上报回执 · 两阶段(device → server)

Topic:`longarch/device/callback/{deviceNo}`

#### 阶段 1:ACK(收到指令后 3 秒内必发)

```json
{
  "taskId": 12345,
  "deviceNo": "ACT-DM01",
  "status": "accepted",
  "msgId": "cb-12345-ack",
  "timestamp": 1720000000500
}
```

作用:让平台快速确认设备**在线、收到指令、准备开始执行**。平台会更新任务的 `device_execution_state = running`。如果 **10 秒**内平台没收到这条,会判"设备未确认接收"的快速失败。

#### 阶段 2:Result(动作完成时发,成功或失败都要发)

**成功**:
```json
{
  "taskId": 12345,
  "deviceNo": "ACT-DM01",
  "status": "success",
  "msgId": "cb-12345-done",
  "timestamp": 1720000300000
}
```

**失败**(比如水流传感器发现没水):
```json
{
  "taskId": 12345,
  "deviceNo": "ACT-DM01",
  "status": "failed",
  "failReason": "no_water_flow",
  "msgId": "cb-12345-done",
  "timestamp": 1720000300000
}
```

**字段说明**:
- `status`:必须是 `accepted` / `success` / `failed` / `network_pending_confirmation` 之一
- `failReason`:自由字符串,失败时给用户看的原因(比如 `no_water_flow`、`valve_stuck`、`over_current`)
- `msgId`:**每条消息全局唯一**,推荐格式 `cb-{taskId}-{seq}`。平台会用它做重放保护
- `timestamp`:UTC 毫秒,平台会校验时间偏差不超过 5 分钟

### 5.3 上报心跳(device → server)

Topic:`longarch/device/heartbeat/{nodeNo}`

**周期**:每 30~60 秒一次。默认超过 **120 秒**无心跳,平台会把设备置 `offline`。

```json
{
  "nodeNo": "EDGE-STM32-01",
  "devices": ["ACT-DM01", "ACT-DM02"],
  "timestamp": 1720000000000
}
```

**字段说明**:
- `nodeNo`:如果你是网关,填网关编号。如果是"单设备就是自己"的场景,把 `nodeNo` 和 `deviceNo` 填成一样即可
- `devices`:该网关名下所有在线执行器的 `deviceNo` 列表。平台会把这些设备的 `network_status` 全部刷新为 `online`

## 6. 对应的软件模块索引

下面这些文件描述了"平台侧是怎么处理你发来的消息的",出问题时能直接对着代码追问题。

| 硬件动作 | 对应的服务端类 | 文件路径 |
| --- | --- | --- |
| 收指令 | `TaskDispatchService`(打包 JSON、发到 MQTT) | `module/task/service/TaskDispatchService.java` |
| 发回执 | `DeviceCallbackListener.handleCallback` | `module/task/mqtt/DeviceCallbackListener.java` |
| 发心跳 | `DeviceCallbackListener.handleHeartbeat` | 同上文件 |
| 发遥测 | `DeviceCallbackListener.handleTelemetry` | 同上文件 |
| 消息校验(重放/大小/身份) | `MqttMessageGuard` | `module/task/mqtt/MqttMessageGuard.java` |
| 心跳超时判离线 | `DeviceHeartbeatStatusScheduler` | `module/task/scheduler/DeviceHeartbeatStatusScheduler.java` |
| ACK/回执超时判失败 | `TaskTimeoutScheduler` | `module/task/scheduler/TaskTimeoutScheduler.java` |
| MQTT 连接配置 | `MqttConfig` + `MqttProperties` | `common/config/MqttConfig.java` |

数据表:
- `actuator_device` — 你的执行器登记表(deviceNo、device_status、network_status、last_heartbeat_at)
- `edge_node` — 网关登记表
- `operation_task` — 每次下发的任务
- `device_execution_log` — 每次执行的原始 payload + 时间戳记录

## 7. 最小实现参考(ESP32 + Arduino)

只做个 demo 时最省事的写法。生产环境请补上 TLS、断线重连、看门狗等健壮性代码。

```cpp
#include <WiFi.h>
#include <PubSubClient.h>
#include <ArduinoJson.h>

const char* WIFI_SSID = "your-wifi";
const char* WIFI_PASS = "your-password";
const char* MQTT_HOST = "192.168.1.100";
const int   MQTT_PORT = 1883;
const char* MQTT_USER = "longarch-esp32";
const char* MQTT_PASS = "your-strong-pw";
const char* DEVICE_NO = "ACT-DM01";

const int   RELAY_PIN = 5;             // 继电器 IO
WiFiClient   net;
PubSubClient mqtt(net);

// 当前正在执行的任务
long currentTaskId    = 0;
long currentEndMillis = 0;

void publishCallback(long taskId, const char* status, const char* failReason = nullptr) {
  StaticJsonDocument<256> doc;
  doc["taskId"]    = taskId;
  doc["deviceNo"]  = DEVICE_NO;
  doc["status"]    = status;
  if (failReason) doc["failReason"] = failReason;
  doc["msgId"]     = String("cb-") + taskId + "-" + status;
  doc["timestamp"] = (long long) time(nullptr) * 1000;

  char buf[256];
  size_t n = serializeJson(doc, buf);
  String topic = String("longarch/device/callback/") + DEVICE_NO;
  mqtt.publish(topic.c_str(), (uint8_t*)buf, n, false);
}

void onCommand(char* topic, byte* payload, unsigned int length) {
  StaticJsonDocument<512> doc;
  if (deserializeJson(doc, payload, length)) return;

  long taskId       = doc["taskId"];
  long expiresAt    = doc["expiresAt"];
  int  durationMins = doc["actionParams"]["durationMinutes"] | 0;

  // 过期就直接报失败, 别执行
  if ((long long)time(nullptr) * 1000 > expiresAt) {
    publishCallback(taskId, "failed", "expired");
    return;
  }

  // 阶段 1: ACK
  publishCallback(taskId, "accepted");

  // 开始执行
  digitalWrite(RELAY_PIN, HIGH);
  currentTaskId    = taskId;
  currentEndMillis = millis() + (long)durationMins * 60 * 1000;
}

void heartbeat() {
  static unsigned long last = 0;
  if (millis() - last < 30000) return;
  last = millis();

  StaticJsonDocument<128> doc;
  doc["nodeNo"]    = DEVICE_NO;                     // 单设备场景:node 就是自己
  JsonArray devs   = doc.createNestedArray("devices");
  devs.add(DEVICE_NO);
  doc["timestamp"] = (long long) time(nullptr) * 1000;

  char buf[128];
  size_t n = serializeJson(doc, buf);
  String topic = String("longarch/device/heartbeat/") + DEVICE_NO;
  mqtt.publish(topic.c_str(), (uint8_t*)buf, n, false);
}

void checkTaskFinish() {
  if (currentTaskId == 0) return;
  if (millis() < currentEndMillis) return;

  // 时间到, 关阀门, 上报成功
  digitalWrite(RELAY_PIN, LOW);
  publishCallback(currentTaskId, "success");
  currentTaskId = 0;
}

void ensureMqtt() {
  while (!mqtt.connected()) {
    if (mqtt.connect(DEVICE_NO, MQTT_USER, MQTT_PASS)) {
      String cmdTopic = String("longarch/device/command/") + DEVICE_NO;
      mqtt.subscribe(cmdTopic.c_str(), 1);
    } else {
      delay(2000);
    }
  }
}

void setup() {
  pinMode(RELAY_PIN, OUTPUT);
  digitalWrite(RELAY_PIN, LOW);
  WiFi.begin(WIFI_SSID, WIFI_PASS);
  while (WiFi.status() != WL_CONNECTED) delay(300);
  mqtt.setServer(MQTT_HOST, MQTT_PORT);
  mqtt.setCallback(onCommand);
  configTime(0, 0, "ntp.aliyun.com");  // 校时, msgId/timestamp 需要
}

void loop() {
  ensureMqtt();
  mqtt.loop();
  heartbeat();
  checkTaskFinish();
}
```

大约 100 行,覆盖了命令接收、两阶段回执、心跳、时长控制、过期保护。

## 8. 联调步骤(建议按顺序)

1. **建设备档案**:在管理后台"设备管理"里给你的硬件登记一条 `actuator_device`,拿到 `device_no`
2. **配 MQTT 账号**:找运维要 Broker 地址、账号密码,先用一个 MQTT 客户端(如 MQTTX)手动订阅 `longarch/device/command/#`,确认能收到消息
3. **手动模拟一次命令**:后端管理后台或 curl 提交一个 pending 任务,观察 MQTT 客户端里是否收到 command payload
4. **硬件订阅 + 打 ACK**:让设备真正订阅 command topic,收到后先只回 ACK 不做任何动作,看小程序里状态是否走到 `running`
5. **加入 Result 回执**:设备执行完真的动作(哪怕就是空转 5 秒 + 亮个 LED)后发 success,看小程序是否显示 `PASSED`
6. **加心跳**:定时发心跳,拔一次网线,看 2 分钟后管理后台里设备是否变 `offline`,以及此时提交任务是否被判 high risk
7. **验证失败路径**:主动发一次 `failed` 回执,看小程序是否显示失败原因;然后模拟不回执,看 5~7 分钟后是否被 stale 扫描判失败

## 9. 常见问题

**Q1:MQTT broker 地址是内网 IP,我在自己网络里连不上怎么办?**
让运维在 Mosquitto 前面加一层 nginx 或直接把 8883(TLS)映射出来。生产环境务必用 TLS + 强密码,不要把 1883 明文暴露公网。

**Q2:`msgId` 一定要全局唯一吗?**
在 replay window(默认 10 分钟)内同一个 `channel:deviceNo:msgId` 唯一即可。用 `cb-{taskId}-{阶段}` 就够,不需要 UUID。

**Q3:两阶段回执如果 ACK 和 Result 之间设备重启了,已经开的阀门怎么办?**
设备启动时应该:①关掉所有 GPIO;②如果本地有掉电前的 `currentTaskId`,直接发一条 `failed` 回执带 `failReason: "device_rebooted"`。这样平台会立刻把任务标失败,同时也会尝试调度下一个任务。

**Q4:我没有传感器,只是纯继电器,能做失败检测吗?**
可以只上报 `success`。想更严一点,加一个电流传感器串在水泵回路上,通电时如果读不到电流(比如保险丝断了),就发 `failed` 带 `failReason: "no_current_detected"`。零件成本几块钱。

**Q5:遥测(sensor telemetry)也走同一套账号密码吗?**
是的,同一个 MQTT 客户端 + 同一套账号,只是往 `longarch/device/telemetry/{sensorNo}` 发即可。传感器数据格式见服务端 `DeviceCallbackListener.handleTelemetry` 的注释。

**Q6:动作时长很短(如喷淋 5 秒),Result 阶段的超时怎么算?**
基线是 `max(30s, durationSeconds + 30s slack)`,不会低于 30 秒。5 秒的动作平台会等 30 秒才判超时,足够容错。
