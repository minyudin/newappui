> 本文件面向**下一个接手的智能体 / 开发者**，用于快速了解 longarch2 的整体结构、已完成工作、当前状态、以及距离正式上线还差什么。

> 最后更新：2026-06-16。对应主干已合并 PR #1（补全部署层 + 修复编译/启动阻塞 + mqtt 无证书开关 + H5 白屏修复）。

## 1. 这是什么项目

**陇上管家 · 智慧农业 / 认养平台**（monorepo，三个子项目）：

| 子项目 | 技术栈 | 角色 | 运行端口 |
| --- | --- | --- | --- |
| `longarch-server` | Spring Boot 3.2.5 / Java 17 / MyBatis-Plus 3.5.6 / MySQL 8 / Flyway / Redis(Redisson) / Sa-Token / MQTT(Paho) / 智谱 GLM | 后端 REST API | 8081 |
| `longarch-admin-next` | React 19 + Vite 7 + TanStack Table + ECharts 6 + Zustand 5（FOLIO 杂志风 UI） | Web 管理后台 | 容器内 nginx 80 → 对外 8080 |
| `miniapp` | Taro 4.2 + React 18（微信小程序，FOLIO 主题） | 认养用户 / 运营端小程序 | 不进 Docker，编译后上传微信平台 |

依赖服务（由 `docker-compose.yml` 一起拉起）：MySQL 3306、Redis 6379、Mosquitto(MQTT) 1883、SRS(媒体服务器) 1935/1985/8880。

### 小程序 17 个页面
4 个登录入口（login / adopter-login / operator-login / guest-login）+ 业务页（home、setup-nickname、share-codes、operator-workbench、adoptions、ai-assist、plot、ai-chat、redeem、task、task-detail、camera、me）。底部 tabBar：运营工作台 / 认养 / AI / 我的。

## 3. 本轮已完成的工作（PR #1，已合并）

> 背景：用户本地代码迭代后推到 longarch2，与历史的 longarch1 仓库**后端已分叉**。要求「以 longarch2 前端为准重新评判真实缺口」。

### 3.1 真实缺口分析结论
把 admin-next + 小程序里**所有 API 调用（约 80 个）**抽出来，逐条对照后端 16 个 Controller：

- **业务功能零缺失** —— 前端每个调用都命中后端已有端点。
- longarch1 里多出来的 3 个模块（**实时 SSE 推送 / 指令 Outbox 可靠下发 / SensorLatestCache 传感器缓存**）**前端根本不调用**，因此**未移植**（移植只会变成无人调用的死代码）。这是经用户确认的决定，不是遗漏。

### 3.2 修复的「能不能上线」级缺口
1. **编译阻塞**：`ProductionStartupGuardTest` 用旧的 1 参构造，而 `ProductionStartupGuard` 已是 `@RequiredArgsConstructor` 的 2 参（`MqttProperties`, `WechatMiniProperties`）。`mvn package` 会编译测试 → 自带 Dockerfile 构建直接失败。已改测试为 2 参构造。
2. **prod 启动崩溃**：`application-prod.yml` 的 mybatis `log-impl` 指向 `org.apache.ibatis.logging.nop.NopImpl` —— 该类在 mybatis 3.5.13~3.5.16 **不存在**，prod profile 启动即 `ClassNotFoundException`。已改为 `org.apache.ibatis.logging.nologging.NoLoggingImpl`。
3. **缺整套部署层**（之前仓库只有 `longarch-server/Dockerfile`）：新增 `docker-compose.yml`、admin `Dockerfile`/`nginx.conf`/`.dockerignore`、`deploy/mosquitto`、`deploy/srs`、`.env.example`，并把 server Dockerfile 的 `EXPOSE` 改成 8081。
4. **MQTT 强制 ssl://**：新增 `mqtt.allow-insecure` 开关（`MqttProperties.allowInsecure`，默认 `false`）。默认仍强制 `ssl://`；显式设 `MQTT_ALLOW_INSECURE=true` 后允许在可信内网用明文 `tcp://`（并打 warn）。
5. **小程序 H5 白屏**：`FolioCorners` / 模块切换遮罩（`position:fixed` 装饰层）原本渲染在 `{children}` 之后，成为 `.taro_router` 下页面节点的后置兄弟，触发 Taro H5 规则 `.taro_router > .taro_page...:not(:last-child){display:none}`，导致所有**非 tab 页**（login/plot/task 等）被整页隐藏白屏。已把装饰层移到 `{children}` 之前。**仅影响 H5**；微信小程序原生导航不受影响。
6. **部署文档**：新增 `部署上线指南.md`。

### 3.3 开 PR 前的验证（docker 全栈）
- `docker compose build`：server + admin 两镜像均构建成功（admin 用 `--frozen-lockfile`，lockfile 同步）。
- `docker compose up -d`：6 个服务全部 Up/healthy；10 条 Flyway 迁移全部应用；`/actuator/health` = `UP`，prod 修复点不再崩溃。
- 经 admin nginx(:8080) 登录后台 → Dashboard（真实统计：1 用户 / 13 地块）、地块页（13 条真实数据 + 分页）、摄像头页均正常渲染 → 证明 `nginx → server → mysql` 全链路通。
- 10 个 admin 读接口（users/plots/cameras/device-overview/adoption-codes/adoption-orders/operation-tasks/sensor-devices/screens/actuator-devices）全部 `HTTP 200 code:0`。

## 5. 距离「正式上线」还差什么（都需用户侧提供，无法离线完成）

| # | 待办 | 说明 |
| --- | --- | --- |
| 1 | 真实微信 `WX_APP_ID` / `WX_APP_SECRET` | prod 守卫强制校验，且需真机 `wx.login` 联调 |
| 2 | 小程序切 `TARO_APP_AUTH_MODE=wechat` 重新编译上传 | 指向正式 HTTPS 域名，提交微信审核 |
| 3 | HTTPS 域名 | 生产登录 Cookie 只走 HTTPS（指南给了 Caddy 自动证书方案） |
| 4 | 生产必填项 | `DASHBOARD_TOKEN`（大屏令牌）、非默认的 MQTT 账号密码（不能是 `longarch`/`longarch123`） |
| 5 | SRS 上线配置 | 把 `SRS_HTTP_FLV_BASE`/`SRS_HLS_BASE` 改成你的域名并经 HTTPS 反代到 `srs:8880`；RTMP 1935 不要对公网开放 |

生产守卫（`ProductionStartupGuard`，仅 prod profile 生效）会逐项检查上述配置，少一项就拒绝启动并打印 `Production configuration invalid: ...`，对照 `部署上线指南.md` 第 4 章即可。

## 6. 给下一个智能体的提示

- **不要再移植 SSE / Outbox / SensorLatestCache**——已确认前端不用，移植即死代码（除非用户后续明确要「实时跳动」体验，那需后端+前端成对做）。
- **不要把 longarch1 和 longarch2 合并**——两套后端已分叉（约 24 个共有文件行为不同；两边都有 `V10` 但内容不同：longarch1 是 `V10__sense_control_optimization.sql`，longarch2 是 `V10__user_nickname_unique.sql`，Flyway 历史无法直接合并）。longarch2 是当前唯一的事实来源。
- **本地起栈**：`cp .env.example .env && docker compose up -d --build`，后台 `http://localhost:8080`，账号 `15675201507` / `admin123456`，接口文档 `http://localhost:8081/doc.html`。
- **admin-next 本地 dev**：`pnpm install && pnpm dev`（Vite，默认 5173/5175），需后端 8081 在跑。
- **小程序 H5 预览**：`pnpm install && pnpm build:h5`（产物在 `miniapp/dist/h5`）。注意 H5 路由是 history 模式，静态服务需 SPA fallback。
- **prod 启动失败先看守卫日志**，对照 `部署上线指南.md` 第 4 章「启动报错对照表」。
